This document provides an in-depth look at the Hivemind Scout component — the web crawling and content management system that enables the Map-Reduce pattern for large web pages.

== Architecture Overview

The scout module has three main components:

. **Crawler** (`scout/crawler.py`) — Wraps `crawl4ai` for web page crawling with batch and recursive modes.
. **Manager** (`scout/manager.py`) — Orchestrates crawl jobs, converts URLs to filesystem paths, saves results.
. **Chunk Cache** (`scout/chunk_cache.py`) — Thread-safe, TTL-based in-memory cache for the Map-Reduce pattern.

[source]
----
┌──────────────┐     ┌──────────────┐     ┌──────────────────┐
│  ScoutManager │ ──▶ │ ScoutCrawler  │ ──▶ │  Filesystem      │
│  (orchestrate)│     │ (crawl4ai)    │     │  (.hivemind/scout)│
└──────────────┘     └──────┬───────┘     └──────────────────┘
                            │
                            ▼
                    ┌──────────────┐
                    │ ChunkCache    │
                    │ (in-memory)   │
                    └──────────────┘
----

== Crawler

The `ScoutCrawler` (link:../scout/crawler.py[scout/crawler.py]) wraps `crawl4ai`'s `AsyncWebCrawler` with:

* **Browser configuration**: Headless Chromium via `playwright`, stealth mode, custom user agent.
* **Content filtering**: `PruningContentFilter` for noise removal (configurable).
* **Markdown generation**: `DefaultMarkdownGenerator` for converting HTML to clean Markdown.
* **Rate limiting**: `RateLimiter` with configurable base delay and exponential backoff.
* **Concurrency control**: `MemoryAdaptiveDispatcher` for parallel crawling with bounded resource usage.

=== Batch Crawling

`crawl_batch(urls)` crawls multiple URLs in parallel:

. Creates a fresh `CrawlerRunConfig` with `stream=True`.
. Sets up a `RateLimiter` and `MemoryAdaptiveDispatcher`.
. Uses `arun_many()` with streaming to yield results incrementally.
. Extracts the best available Markdown (prefers `markdown_v2.fit_markdown`).

=== Recursive Crawling

`crawl_recursive(url, max_pages, max_depth, ...)` performs BFS-based deep crawling:

. Uses `BFSDeepCrawlStrategy` with configurable `max_depth` and `max_pages`.
. Applies a `FilterChain` for URL inclusion/exclusion patterns.
. Supports `stay_in_path` mode (only follow URLs under the seed URL).
. Yields results incrementally via async generator.

=== Markdown Extraction

The `_extract_best_markdown()` helper prefers the cleanest available Markdown:

[source,python]
----
def _extract_best_markdown(result):
    content = result.markdown or ""
    if hasattr(result, "markdown_v2") and result.markdown_v2:
        mv2 = result.markdown_v2
        if hasattr(mv2, "fit_markdown") and mv2.fit_markdown:
            content = mv2.fit_markdown
    return content
----

== Manager

The `ScoutManager` (link:../scout/manager.py[scout/manager.py]) orchestrates crawl jobs and manages output.

=== URL to Path Conversion

Each crawled URL is saved to a structured filesystem path:

[source,python]
----
def _url_to_path(self, url: str) -> Path:
    parsed = urlparse(url)
    domain = parsed.netloc.replace(".", "_")
    url_path = parsed.path.strip("/") or "index"
    if parsed.query:
        query_hash = md5(parsed.query.encode()).hexdigest()[:6]
        url_path = f"{url_path}_{query_hash}"
    return Path(domain) / f"{url_path}.md"
----

Example: `https://docs.example.com/guide/getting-started` → `docs_example_com/guide/getting-started.md`

=== URL Range Expansion

URLs containing range placeholders like `{1..100}` are expanded:

[source,python]
----
# Input
"https://docs.example.com/page/{1..5}"

# Output
["https://docs.example.com/page/1",
 "https://docs.example.com/page/2",
 "https://docs.example.com/page/3",
 "https://docs.example.com/page/4",
 "https://docs.example.com/page/5"]
----

=== Skip Existing

When `skip_existing` is enabled, the manager scans the output directory for previously crawled URLs by reading the `source_url` field from YAML frontmatter in saved Markdown files.

=== Output Format

Each crawled page is saved as a Markdown file with YAML frontmatter:

[source,markdown]
----
---
source_url: https://docs.example.com/page
seed_url: https://docs.example.com
---

# Page Title

Content...
----

=== Chunked Processing

For large batch crawls, URLs are processed in chunks of 100 to prevent memory/task overhead:

[source,python]
----
chunk_size = 100
for i in range(0, len(target_urls), chunk_size):
    chunk = target_urls[i:i + chunk_size]
    async for page_url, content in self.crawler.crawl_batch(chunk):
        self._save_page(page_url, content)
----

== Chunk Cache (Map-Reduce Pattern)

The `ChunkCache` (link:../scout/chunk_cache.py[scout/chunk_cache.py]) enables the Map-Reduce pattern for the `scout_urls` MCP tool.

=== How It Works

. **Map**: When `scout_urls` is called with `mode="full"` or `mode="toc"`, the URL is crawled and the content is chunked via `MarkdownChunker`. The chunks are stored in the in-memory cache keyed by a SHA-256 hash of the URL.
. **Reduce (TOC)**: The agent requests a Table of Contents. The cache returns section names, heading levels, line ranges, and estimated token counts — without re-crawling.
. **Reduce (Sections)**: The agent requests specific sections by name or index. The cache returns only the requested chunks.

=== Cache Details

* **Storage**: `Dict[str, tuple[List[Chunk], float]]` — maps URL hash to (chunks, timestamp).
* **TTL**: 10 minutes (configurable via `DEFAULT_TTL`).
* **Thread safety**: All operations are protected by a `threading.Lock`.
* **Eviction**: Expired entries are removed on every `get()` call.
* **Singleton**: Accessed via `get_chunk_cache()` which returns a process-wide singleton.

=== TOC Format

The Table of Contents shows:

[source]
----
## Table of Contents: https://example.com

| # | Section | Level | Lines | ~Tokens |
|---|---------|-------|-------|---------|
| 0 | (preamble) | intro | 1-3 | 12 |
| 1 | Installation | H2 | 4-50 | 450 |
| 2 | Configuration | H2 | 51-120 | 800 |
| 3 | Usage | H2 | 121-200 | 600 |
----

=== Section Matching

When filtering sections, matching is case-insensitive substring against header text, plus exact numeric index match:

[source,python]
----
for term in terms:
    if term == idx_str or term in label:
        selected.append(chunk)
        break
----

This means `sections=["Installation"]` matches a section titled "Installation", and `sections=["2"]` matches the third chunk (0-indexed).

== Chunk Formatter

The `chunk_formatter` module (link:../scout/chunk_formatter.py[scout/chunk_formatter.py]) provides two formatting functions:

* `format_toc(chunks, url)` — Generates the Table of Contents markdown table.
* `format_sections(chunks, url, section_names)` — Returns verbatim content for matching sections.

Token estimation uses a conservative 4 characters per token ratio.

== CLI Integration

The scout is accessible via the CLI:

[source,bash]
----
# Crawl specific URLs
hivemind scout crawl https://docs.example.com

# Recursive crawl
hivemind scout crawl https://docs.example.com --recursive

# With options
hivemind scout crawl https://docs.example.com \
    --recursive \
    --max-pages 100 \
    --stay-in-path \
    --include "*/guide/*" \
    --concurrency 10 \
    --delay 2.0
----

See link:cli.adoc[CLI Command Reference] for full option details.

== MCP Integration

The scout powers two MCP tools:

* `search_web` — Searches the web via DuckDuckGo (or configured backend), returns title/URL/snippet.
* `scout_urls` — Crawls URLs and returns content with Map-Reduce support.
* `deep_research` — Combines search + crawl in a single call.

All three tools are defined in link:../server/server.py[server/server.py] and delegate to the scout module for crawling.

== Requirements

The scout module requires optional dependencies:

[source,bash]
----
# Install scout extras
uv sync --extra scout

# Install Chromium for Playwright
playwright install chromium
----

Without these, the `scout_urls` and `deep_research` MCP tools return a helpful error message with installation instructions.
